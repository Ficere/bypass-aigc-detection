# BypassAIGC 系统架构参考

## 技术架构

```
用户输入全文
    │
    ▼
┌─────────────────────┐
│   智能分段引擎        │  按 \n 分割，超长段按句号再分割
│   (max 500 字/段)    │  短段落(<15字) 标记为标题直接跳过
└─────────┬───────────┘
          │
          ▼
┌─────────────────────────────────────────────┐
│            第一阶段：论文润色 (Polish)          │
│                                              │
│  对每个段落：                                  │
│  1. 构建 messages = [history] + [system] + [user] │
│  2. system = polish_prompt + 防注入提示         │
│  3. user = 当前段落原文                        │
│  4. 调用 LLM 获取润色结果                      │
│  5. 将结果追加到 history (role=assistant)       │
│  6. 如果 history 字符数 > 阈值，压缩历史       │
└─────────────────┬───────────────────────────┘
                  │
                  ▼
┌─────────────────────────────────────────────┐
│          第二阶段：原创性增强 (Enhance)         │
│                                              │
│  对每个段落：                                  │
│  1. 输入 = 第一阶段的润色结果                  │
│  2. 构建 messages = [history] + [system] + [user] │
│  3. system = enhance_prompt + 防注入提示        │
│  4. 调用 LLM 获取增强结果                      │
│  5. 将结果追加到 history (role=assistant)       │
│  6. 如果 history 字符数 > 阈值，压缩历史       │
└─────────────────┬───────────────────────────┘
                  │
                  ▼
          最终输出（拼接所有段落）
```

## 上下文管理策略

### 历史会话传递

- 每个已处理段落的 LLM 输出作为 `assistant` 消息加入 history
- 标题段落不参与历史上下文
- 后续段落在处理时能"看到"之前段落的处理风格，保证全文一致性

### 历史压缩机制

当 history 中的中文字符总数超过阈值（默认 5000 字）时：

1. 保留最近 2-3 条消息作为风格参考
2. 将所有 assistant 和 system 消息的内容合并
3. 调用 compression LLM 压缩为摘要（不超过原内容的 30%）
4. 用一条 system 消息替换整个 history："之前处理的段落摘要：{compressed}"

### 思考标签过滤

某些模型（DeepSeek、o1 等）会输出 `<think>...</think>` 标签，系统会：
- 在流式输出中实时检测和过滤思考标签
- 在非流式输出中用正则表达式移除

## 处理模式

| 模式 | 第一阶段 | 第二阶段 | 适用场景 |
|------|---------|---------|---------|
| paper_polish_enhance | Polish | Enhance | 论文（效果最好） |
| paper_polish | Polish | — | 论文（改动小） |
| paper_enhance | — | Enhance | 论文（直接增强） |
| emotion_polish | Emotion | — | 感情文章 |

## 推荐模型

项目推荐使用 Gemini 2.5 Pro，原因：
- 性能好且成本低
- 通过 OpenAI 兼容格式的代理服务调用
- 默认禁用流式输出以避免 Gemini API 的阻止错误
- 支持 thinking mode (reasoning_effort: high)

## 并发与错误处理

- 最大并发用户数可配置（默认 5）
- 超出并发限制的请求进入队列等待
- 段落处理失败时记录失败位置，支持断点续传
- 不可重试错误（权限拒绝、认证失败）直接抛出
- 可重试错误（速率限制）尝试降级重试（移除 reasoning_effort）
- API 请求间隔可配置（默认 6 秒），避免触发速率限制
